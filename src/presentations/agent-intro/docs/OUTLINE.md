# 演讲大纲：从 API 调用到 Agent Harness

## 基本信息

| 项目 | 说明 |
|------|------|
| 受众 | 全公司 All-Hands（工程师 / 产品 / 运营 / 设计 / 管理层混合） |
| 时长 | 45-60 分钟 |
| 风格 | 叙事演进 + 代码渐进展示 |
| 张数 | ~23 张 |
| 关联 | 技术深讲版见 `agent-slides`（12 个工程主题） |

---

## 核心叙事逻辑

1. **从听众自己用过的 Agent 出发**（互动），引出 5 种实现形态
2. **讲解这 5 种形态的演进历史**，推导出 Harness 的必要性
3. **重点：Harness 为什么比模型更关键**
4. **从最简单的 Agent Loop 开始，逐步加功能**，亲手搭起完整 Agent
5. **收尾：我们的机会在哪里，各岗位怎么行动**

**叙事张力**：「你们提到的那些 Agent，其实处于不同的工程成熟度阶段——今天我们把这张地图展开」

### Slide 4：形态 0 — 裸 API 调用

**左栏：代码**
```typescript
const response = await client.messages.create({
  model: "claude-opus-4-6",
  messages: [{ role: "user", content: userInput }]
});
// 结束，这次对话永远消失
```

**右栏：时序图**
```
👤 用户              🤖 LLM
  │                    │
  │── 发送消息 ────────→│
  │                    │ (思考)
  │←─ 返回回复 ────────│
  │                    │
  ✗ 连接断开，LLM 不再记得这次对话
```
- **视觉重点**：连接线只有一来一回，然后打上 ✗，表示「此次对话就此消失」
- **类比文字**（图表下方）：像发一条没有留存的语音消息

### Slide 5：形态 1 — 加了记忆（Messages 数组）

### Slide 6：形态 2 — 基于工作流的节点 Agent

**左栏：伪代码 / 流程定义**

```
         患者描述症状
               │
        ┌──────▼──────┐
        │  LLM 判断   │
        └──┬───┬───┬──┘
           │   │   │
```

### Slide 7：形态 3 — 有了 Tools（工具调用）

**左栏：代码**（高亮工具定义部分）
```typescript
const tools = [
  {
    name: "query_account",       // ← 新增：工具定义
    description: "查询玩家账号记录",
    input_schema: { ... }
  }
];
const response = await client.messages.create({
  model, messages,
  tools,                         // ← 新增：把工具交给 LLM
});
// LLM 自己决定要不要调用，调用哪个
```

**右栏：交互时序图**
```
👤 用户        🤖 LLM         🔧 工具（数据库/API）
  │              │                    │
  │── 发申诉 ───→│                    │
  │              │── 调用query_account→│
  │              │                    │ (查询)
  │              │←─ 返回账号记录 ────│
  │              │ (分析结果，决策)    │
  │←─ 回复结论 ──│                    │
```
- **视觉重点**：LLM 与外部系统之间多了一条双向箭头，LLM「伸出手」触达外部世界
- **底部标注**：「有了工具，LLM 从只会说话变成能干活」
- **突出缺陷**：最后加一行「但这一次就结束了，它不会主动说『我还没做完，继续』」

### Slide 8：形态 4 — 有了循环（Agent Loop）

**左栏：代码**（高亮 while 循环）
```typescript
const messages = [{ role: "user", content: userInput }];

while (true) {                           // ← 新增：循环
  const response = await client.messages.create({
    model, tools, messages
  });
  if (response.stop_reason === "tool_use") {
    const results = await executeTools(response.content);
    messages.push(...);                  // 工具结果塞回去
  } else {
    return response.content;            // LLM 自己说「完了」
  }
}
```

**右栏：循环状态机图**
```
        ┌─────────────────────────────────┐
        │             循环体              │
        │                                 │
  输入 ─→ [LLM 决策] ──需要工具──→ [执行工具]
        │      │                      │   │
        │      │←──── 工具结果 ────────┘   │
        │      │                           │
        │    任务完成？                     │
        │    YES ↓                         │
        └──→ [输出结果]   NO → 继续循环 ──→┘
```
- **视觉重点**：循环箭头形成闭环，「任务完成」是唯一出口
- **动画**：循环箭头持续脉冲，直到 Fragment 触发「任务完成」分支点亮
- **底部标注**：「整个核心逻辑不到 20 行，这就是 Agent 的心脏」

### Slide 9：Workflow vs Agent — 该用哪个？
| | Workflow | Agent |
|---|---|---|
| 路径 | 由代码预先写死 | 由 LLM 动态决定 |
| 适合 | 路径可预测、规则清晰 | 路径无法穷举 |
| 优势 | 确定性强，易测试 | 灵活，能处理未知情况 |
| 风险 | 遇到例外直接失效 | 行为不可完全预期 |

> 两者没有高下，选错工具比选错模型代价更大。

### Slide 10：形态 5 — 复合系统（豆包这类）

**右栏（主视觉）：分层架构图**
```
┌─────────────────────────────────────────┐
│              产品界面层                  │  ← 用户看到的
│         （对话框 / App / 小程序）         │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│              路由 / 编排层               │  ← 工作流节点
│    判断意图 → 分发到不同子系统            │
└───┬──────────────┬──────────────────────┘
    │              │
┌───▼───┐    ┌─────▼──────────────────────┐
│  知识  │    │         Agent Loop          │  ← 核心执行
│  库   │    │  LLM + Tools + 记忆         │
└───────┘    └─────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│              工具 / API 层               │  ← 真实执行
│   查数据库  发消息  改配置  调外部服务    │
└─────────────────────────────────────────┘
```
- **视觉重点**：每一层用不同颜色标注，Fragment 从下到上逐层出现
- **左栏标注**：「这 4 层每一层都对应前面讲过的某种形态」
- **底部结论**：「越复杂 → 出问题时越难定位是哪一层的责任」

### Slide 11：问题来了——Loop 本身不够

**左栏：Loop 运行时遇到的真实问题**
```
while (true) {
  const response = await client.messages.create(...);

  // ❌ 工具调用超时了，怎么办？重试？重试几次？
  // ❌ LLM 输出了错误格式，工具解析失败，然后呢？
  // ❌ 循环了 50 次还没完，是卡住了还是在干活？
  // ❌ 这次结果对不对？和上次比变好了还是变差了？
  // ❌ 出了问题，哪一步出的？日志在哪？
}
```

**右栏：失控可视化**
```
正常预期：                    现实：
                              第1次 ✓
输入 → Loop → 输出            第2次 ✓
   （干净，可控）             第3次 ✗ 工具超时
                              第4次 ✗ 无限重试 → 💸💸💸
                              第5次 ? 输出对不对？不知道
                              第6次 ✗ 崩了，从哪恢复？
```
- **视觉重点**：左侧「理想」简洁，右侧「现实」一片混乱，对比强烈
- **动画**：右侧每个问题 Fragment 依次出现，逐渐「染红」整个 Loop 图
- **过渡语**：「这就是为什么光有 Loop 不够——我们需要 Harness」

## Part 2 · 为什么 Harness 比模型更关键（5 张，~12 min）

### Slide 12：什么是 Harness？
围绕 Agent Loop 的四件套基础设施：

| 组件 | 作用 | 类比 |
|------|------|------|
| **验收基线** | 知道什么算做对了 | 飞行前检查清单 |
| **执行边界** | 知道什么不该做 | 飞行禁区 |
| **反馈信号** | 知道哪步出错了 | 飞行仪表盘 |
| **回退手段** | 出错了能恢复 | 自动驾驶失效后的手动接管 |

### Slide 13：没有 Harness 的 Agent 会怎样
**对比演示**：
- 工具调用失败 → 无 Harness：无限重试，烧光 token；有 Harness：触发回退机制
- 上下文被污染 → 无 Harness：越走越偏，无从发现；有 Harness：反馈信号告警
- 新版比旧版差 → 无 Harness：不知道；有 Harness：验收基线告诉你

> **工程洞察**：Harness 质量的影响比模型质量更大。大多数「模型不稳定」的问题，根源在 Harness 缺失。

### Slide 14：Claude Code / OpenClaw 为什么能稳定工作
这类工具之所以好用，不是因为模型更强，而是因为 Harness 做得扎实：
- **任务边界**：知道自己能做什么，不能做什么
- **Worktree 隔离**：每个任务在独立环境，不污染主分支
- **人工确认节点**：高风险操作暂停等待确认
- **完整 Trace**：每步操作可追溯、可回放

### Slide 15：「换个更强的模型」为什么没用
绝大多数 Agent 失控，根源在这里：
- 工具描述不准确（加反例可从 53% → 85% 准确率）
- 上下文设计不当（信息太多 = 信息等于没有）
- 没有验收基线（不知道好不好，就乱调）

> 换模型不解决这些问题。类比：给一个优秀员工一本错误的操作手册，换个更聪明的员工结果更糟。

### Slide 16：核心结论
1. Agent 的**能力上限** = 工具集 + 数据访问权限
2. Agent 的**稳定性** = Harness 质量，不是模型质量
3. Agent 的**风险** = 执行边界不清晰

## Part 3 · 现在我们能做什么（5 张，~12 min）

### Slide 17：我们的场景在哪
| 场景 | 业务价值 | 实施难度 | 优先级 |
|------|---------|---------|------|
| 游戏客服（申诉/补偿） | 高 | 中 | ⭐⭐⭐ |
| 内容审核辅助 | 高 | 中 | ⭐⭐⭐ |
| 内部效率工具 | 中 | 低 | ⭐⭐⭐ |
| 精准推送 | 中 | 低 | ⭐⭐ |
| 智能 NPC | 高 | 高 | ⭐ |
| 数据分析助手 | 中 | 中 | ⭐⭐ |

### Slide 18：判断一个场景——4 个问题
1. **路径是否可穷举？** 能 → 用 Workflow 更好
2. **犯错代价多高？** 高 → 需要更多人工确认节点
3. **需要的工具是否可建？** 数据可访问性决定能力上限
4. **量是否够？** 量小 → ROI 不成立

### Slide 19：完整例子——游戏客服 Agent
```
玩家申诉「我被误封了」
  ↓ [工具调用] 查询玩家行为记录 + 封号原因
  ↓ [工具调用] 比对规则库，判断是否误封
  ↓ [工具调用] 调用解封 API + 发放补偿道具
  ↓ 自动回复玩家，记录处理结果
```
每步标注：调用了什么工具 + Harness 的哪个组件在保护它

### Slide 20：从 0 到 1 的正确方式
**❌ 错误路径**：找最复杂的场景，追求一步到位 → 上线即崩

**✅ 正确路径**：
1. 找高频 + 规则清晰 + 容错大的小场景
2. 快速出第一版（第一版一定是烂的，正常）
3. 建测试基线（知道好不好）
4. 迭代，而不是重写

### Slide 21：各岗位现在能做什么
| 岗位 | 现在就能做 |
|------|---------|
| 工程师 | 用 Cursor / GitHub Copilot 感受 Agent，积累 Prompt 设计经验 |
| 产品 / 运营 | 识别你负责流程里「高频 + 规则清晰」的任务 |
| 管理层 | 支持一个小场景试点，允许第一版失败，这是正常成本 |

## 图表实现方案

五种形态的图表全部用 **CSS + React 内联组件**实现，无需外部图片资源：

| 形态 | 图表类型 | 实现方式 |
|------|---------|---------|
| 形态 0：裸 API | 时序图（单来回）| CSS flex 行，箭头用 `→ ←`，末尾 ✗ 标记 |
| 形态 1：记忆 | 消息堆叠图 | CSS flex-wrap，消息气泡随 Fragment 递增 |
| 形态 2：工作流 | 决策树/流程图 | CSS grid + 连接线，菱形用 `transform: rotate(45deg)` |
| 形态 3：Tools | 交互时序图（三方）| CSS flex 三列，双向箭头，外部系统用不同背景色 |
| 形态 4：Loop | 循环状态机 | CSS 环形布局 + `@keyframes` 脉冲箭头动画 |
| 形态 5：复合 | 分层架构图 | CSS flex 纵列，每层不同颜色，Fragment 从底层往上出现 |

**布局策略**：
- 每张幻灯片：左 40% 代码 + 右 60% 图表（或图表占全屏，代码作为小块叠加）
- 非技术受众的视线自然落在右侧大图
- 技术受众看左侧代码，图表起「确认理解」作用

**动画联动**：
- 代码高亮（新增行变色）与图表新增节点同步出现
- 同一 Fragment 触发左侧代码高亮 + 右侧图表对应元素的 `fade-up`

## 代码展示策略

- **形态 0-4**：每张幻灯片的代码是前一张的「增量版本」，新增内容用高亮色标出
- **非技术受众**：不要求看懂代码，看懂「加了什么新东西」即可
- **技术受众**：可以直接从代码获得工程信息

示例（形态 0 → 形态 1 的变化）：
```diff
  const response = await client.messages.create({
    model: "claude-opus-4-6",
+   messages: history,  // ← 这就是「记忆」
  });
+ history.push(...);   // ← 把这次对话记下来
```
